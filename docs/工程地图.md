# Serene Ledger 工程地图（简明版）

## 1) 项目概览
- 框架: Next.js 15（App Router）+ React 19 + TypeScript
- 样式: Tailwind CSS v4（`@theme inline` 写在 `src/app/globals.css`）
- 数据与鉴权: Supabase（Auth + PostgreSQL）
- 渲染方式: 主要为客户端组件（大部分页面和组件都 `use client`）

## 3) 状态管理（State Management）

### 3.1 总体策略
- 没有 Redux/Zustand/Context 全局业务状态。
- 采用「页面本地 `useState` + 领域 Hook」模式：
  - 领域 Hook 负责拉取/转换/提交数据；
  - 页面与组件只保留 UI 交互状态（选中项、弹窗开关、编辑态）。

### 3.2 关键 Hook 状态
- `useAuth`
  - 状态: `user`, `loading`
  - 来源: `supabase.auth.getUser()` + `onAuthStateChange`
  - 行为: `signOut()`

- `useClients`
  - 状态: `clients`, `loading`, `allTags`
  - 读取: `clients` + `sessions` + `event_clients` 并做前端聚合
  - 行为: `addClient`, `updateClient`, `addSession`, `updateSession`, `deleteTag`, `refetch`

- `useSchedule`
  - 状态: `items`, `loading`
  - 读取: `sessions` + `events` + `clients` + `event_clients`，拼成统一 `ScheduleItem[]`
  - 行为: `addEvent`, `getItemsForDate`, `refetch`

- `useStatistics`
  - 状态: `stats`, `loading`
  - 读取: `sessions`, `events(type=supervision)`, `clients`
  - 计算: 总量、状态分布、近 6 个月趋势、月度概览

### 3.3 组件本地状态（典型）
- 页面层: `selectedDate`、`viewMode`、`selectedId`、`showModal`
- 详情/弹窗: 表单字段、`editing`、`submitting`

## 4) 数据流（Data Flow）

## 4.1 鉴权流
1. 进入非 `/login` 路由 -> `AppShell` 包裹 `AuthGuard`
2. `AuthGuard` 通过 `useAuth` 获取用户
3. 未登录则 `router.push("/login")`
4. 登录页 `supabase.auth.signInWithPassword/signUp` 成功后进入主页

## 4.2 个案管理流（`/`）
1. `HomePage` 调用 `useClients()` 拉取聚合数据
2. 左侧列表按 `statusFilter` 前端过滤
3. 选中个案后展示 `ClientDetail`
4. 在 `ClientDetail` 内进行：
   - 新增咨询 -> `AddSessionModal` -> `onAddSession` -> `useClients.addSession`
   - 编辑咨询 -> `SessionDetail` -> `onUpdate` -> `useClients.updateSession`
   - 编辑个案/改状态 -> `useClients.updateClient`
5. 成功后通过 `fetchClients()` 全量刷新 `clients`

## 4.3 日程流（`/schedule`）
1. `SchedulePage` 同时使用 `useSchedule()` 与 `useClients()`
2. `useSchedule` 统一产出咨询/督导/其他日程项
3. 日/周视图展示后，点选某项进入 `DetailPanel`
4. 新建日程：
   - 咨询: 走 `onSubmitConsultation`（最终写入 `sessions`）
   - 督导/其他: 走 `onSubmitEvent`（写入 `events`，督导可写 `event_clients`）
5. 关闭弹窗时触发 `refetch()` 更新日程列表

## 4.4 统计流（`/stats`）
1. `useStatistics` 拉基础表数据
2. 前端计算总量、状态分布、月度趋势
3. 页面通过 `monthIndex` 切换月度卡片显示

## 5) 关键模块职责（按“谁负责什么”）
- `src/app/layout.tsx`
  - 站点元信息、全局样式入口、挂载 `AppShell`

- `src/components/app-shell.tsx`
  - 判断登录页是否跳过壳层；其余页面统一包裹导航和鉴权

- `src/components/auth-guard.tsx`
  - 登录态门禁；处理加载态和重定向

- `src/hooks/useClients.ts`
  - 个案域核心数据层（聚合、映射、CRUD、刷新）

- `src/hooks/useSchedule.ts`
  - 日程域核心数据层（跨表拼装成统一日程项）

- `src/hooks/useStatistics.ts`
  - 统计域计算层（聚合统计口径）

- `src/app/schedule/*`
  - 日程页面的展示逻辑层（时间维度视图与交互）

- `src/components/*modal*.tsx`
  - 录入与编辑的表单交互层（不直接持久化，调用上层传入方法）

## 6) 当前架构特征与维护要点
- 优点:
  - 结构直观，页面 -> Hook -> Supabase 的链路清晰；
  - 无复杂全局状态，理解门槛低；
  - 类型定义集中在 `src/types/index.ts`，便于统一演进。

- 注意点:
  - 数据聚合主要在前端完成，数据量增大后性能与请求体积会成为瓶颈。
  - 多个页面各自拉取并维护数据（如 `useClients` 在主页/日程页分别调用），存在重复请求。
  - 错误处理目前偏轻（多数写操作只返回 `boolean`），可逐步补充错误原因与用户反馈。

---

## 7) 一句话理解
这是一个“客户端主导、按业务域拆 Hook”的 Next.js + Supabase 应用：页面负责交互，Hook 负责取数与写库，组件负责展示与表单录入。
